## 二、各层详解

### 2.1 接入层：cc-connect

**职责**：把各种消息平台的消息统一收进来，把Agent的回复发出去。

**已验证的通道**：

| 通道 | 状态 | 模式 | 说明 |
|------|------|------|------|
| 微信 (weixin/ilink) | ✅ 已通 | token认证 | 个人微信消息收发 |
| 企业微信 (wecom) | 🔧 待完成 | callback | 需要nginx反代(yixiubx.com) |
| 飞书 (feishu) | ✅ 已通 | app_id/secret | 内部协作备用 |

**关键技术点**：
- 每个Agent对应一个企微bot实例
- callback模式：企微把群消息POST到我们的服务器 → 我们处理后回复
- 需要nginx在yixiubx.com做反向代理，把443端口的 /wecom/ 转发到本地cc-connect的8900端口

**配置模板已就绪**：
- nginx配置：`~/Desktop/yixiubx-nginx-wecom.conf`
- cc-connect配置：`~/Desktop/cc-connect-callback.toml`

### 2.2 Agent调度层：Router

**职责**：决定哪条消息给哪个Agent处理。

**核心功能**：

#### 消息路由
```
群消息进来
    ↓
识别@了哪个Agent（或关键词匹配）
    ↓
分发给对应Agent处理
    ↓
Agent回复 → 以对应bot身份发到群里
```

**路由策略（待测试确定）**：

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| @指定Agent | 用户手动@核保助手 | 明确，但需要用户记住有哪些Agent |
| 关键词匹配 | 系统识别"核保"、"理赔"等关键词自动分发 | 自然，但可能误判 |
| 调度Agent | 一个主Agent先判断问题类型，再转发 | 智能，但多一次LLM调用 |
| 混合模式 | 优先@，没@走自动识别 | 灵活 |

#### 并发控制
```
多群同时提问时：
    ↓
消息队列排队（按优先级）
    ↓
限制同时处理的并发数（如最多同时回复5个群）
    ↓
每个回复加随机延迟（3-10秒，模拟真人打字）
```

#### 频率控制
```
单Agent限制：
    · < 10条/分钟 → 安全
    · 10-20条/分钟 → 加延迟
    · > 20条/分钟 → 排队等待

多Agent分散：
    · 同类问题多时，多个同类Agent分担
```

### 2.3 Agent执行层：Claude Code + Skills

**职责**：每个Agent是一个独立的Claude Code实例，挂载对应岗位的Skills。

#### Agent架构
```
Agent = Claude Code 实例
      + 岗位Skill（专业知识）
      + 个性化配置（名字、风格）
      + 上下文记忆（对话历史）
```

#### Skill设计（以核保Agent为例）
```
核保Skill/
├── SKILL.md              # Skill入口，定义触发条件和行为
├── knowledge/
│   ├── 核保规则.md        # 通用核保规则
│   ├── 健康告知标准.md    # 各类疾病的核保结论
│   ├── 产品费率表.md      # 各产品费率计算逻辑
│   └── 合规要求.md        # 监管红线、必查项
├── workflow/
│   ├── 标准核保流程.md    # 一步步怎么判断
│   └── 疑难case处理.md   # 复杂情况的处理逻辑
└── examples/
    └── 优秀案例.md        # 老核保员的经典判断
```

#### 个性化配置
```json
{
  "agent_id": "uw_001",
  "name": "小李核保助手",
  "avatar": "avatar_url",
  "style": {
    "tone": "专业但亲和",
    "verbosity": "简洁为主，必要时详细",
    "emoji": "适度使用"
  },
  "owner": "业务员A的企微ID",
  "company": "XX保险公司"
}
```

#### 多Agent在同一群的协作
```
群里：业务员A + 客户张三 + 核保助手 + 理赔助手 + 客服助手

客户问："我的车昨天被刮了怎么办"
    → Router识别为理赔问题 → 理赔助手响应

业务员问："这个客户的甲状腺结节能不能标体"
    → Router识别为核保问题 → 核保助手响应
```

### 2.4 人机协作机制（Agent答不了怎么办）

**核心原则**：Agent不确定的问题，带给本尊（业务员）确认后再回复。本尊看到完整上下文，修改或确认后Agent代发。

#### 置信度分级

Agent在每次回答前内部评估置信度（不展示给客户）：

| 置信度 | 判断标准 | 行为 |
|--------|----------|------|
| **HIGH** | 知识库能找到明确答案 | 直接回复到群里 |
| **MEDIUM** | 知识库有相关内容但需推断 | 回复缓冲 + 私聊本尊确认 |
| **LOW** | 知识库完全没有覆盖 | 回复缓冲 + 私聊本尊确认 |

#### 完整流程

```
【客户群】
客户张三：我这个甲状腺结节会影响理赔吗？
Agent：好的，我帮您确认一下，稍后给您准确回复。
         ↓（同时触发私聊通知）

【私聊：本尊 ↔ Agent】
Agent发送带上下文的通知：
┌──────────────────────────────────┐
│ 🔔 需要您确认                     │
│                                    │
│ 📍 群：张三的服务群                │
│ 👤 客户：张三（重疾险 #2023xxx）   │
│                                    │
│ 📋 最近对话：                      │
│ 张三：我想问一下理赔的事            │
│ Agent：好的，请问是什么情况？       │
│ 张三：我上个月查出来甲状腺结节      │
│ Agent：了解，请问是几级的？         │
│ 张三：3级                         │
│ 张三：这个会影响理赔吗？  ← 当前问题│
│                                    │
│ 🤖 我的草稿：                     │
│ "甲状腺结节3级以上会影响理赔..."   │
│ 置信度：低（知识库未覆盖此场景）    │
│                                    │
│ 请回复：                          │
│ 1 = 发送草稿                      │
│ ✏️ 直接回复修改后的内容           │
│ 3 = 跳过不答                      │
└──────────────────────────────────┘
         ↓
本尊回复修改内容："甲状腺结节3级以上会影响，
                  具体要看投保时的健康告知记录"
         ↓

【客户群】
Agent：张先生，甲状腺结节3级以上会影响理赔，
      具体要看您投保时的健康告知记录。
```

#### 上下文摘要机制

Agent在每个客户群实时维护对话摘要，通知本尊时打包发送：

```
上下文摘要包含三部分：

1. 客户基本信息
   - 姓名、保单号、险种、投保时间
   - 首次对话时获取，后续自动更新

2. 对话摘要（滚动更新）
   - 最近5-10轮对话的压缩版
   - 关键信息提取（疾病名称、条款编号、金额等）
   - 如果对话很长，AI先做摘要压缩

3. 当前问题
   - 最新一条消息原文
   - Agent的草稿回答 + 置信度评估
```

#### 私聊通道

| 方案 | 说明 | 建议 |
|------|------|------|
| 企微bot私聊 | Agent的企微bot给业务员发私聊 | ✅ 首选，业务员不用切App |
| 单独内部群 | 业务员 + Agent的二人小群 | 备选 |
| 微信私聊 | 通过ilink通知业务员 | 业务员不在企微时使用 |

#### 超时处理

```
通知本尊 → 开始计时
    ↓
5分钟无回复 → 再通知一次（提醒）
    ↓
15分钟无回复 → 群里告知客户"已转人工处理，会尽快回复"
    ↓
本尊后来回复了 → Agent补发到群里
```

#### 多问题合并

```
客户连续问了多个Agent答不了的问题：
  Q1 → 缓冲 + 加入待确认队列
  Q2 → 缓冲 + 加入待确认队列
  Q3 → 缓冲 + 加入待确认队列

私聊里合并通知本尊，一次确认多个：
┌──────────────────────────────┐
│ 🔔 有3个问题需要您确认         │
│                                │
│ Q1: xxxxx                      │
│ 草稿: xxxxx                    │
│                                │
│ Q2: xxxxx                      │
│ 草稿: xxxxx                    │
│                                │
│ Q3: xxxxx                      │
│ 草稿: xxxxx                    │
│                                │
│ 逐条回复"1/修改/3"即可        │
└──────────────────────────────┘
```

#### 知识沉淀（飞轮入口）

```
每次本尊修改过的回答 → 自动记录：
  · 原问题
  · Agent草稿
  · 本尊修正
  · 对话上下文

这些是飞轮最有价值的燃料：
  定期整理 → 封装为新Skill → Agent下次同类问题直接能答
```

### 2.5 数据层

#### 知识库
- **通用知识库**：保险行业通用规则（所有客户共享）
- **客户专属库**：某保险公司的产品条款、内部规则（数据隔离）
- **RAG检索**：Agent回答时实时检索相关知识片段

#### 对话历史
- 每个群维护独立的对话上下文
- 支持多轮对话（如核保追问：有没有其他病史？→ 甲状腺3级 → 建议除外）
- 长期记忆：记住该客户的保单信息、历史咨询记录

#### 用户配置
- 每个Agent的个性化配置（名字、头像、风格）
- 每个公司的定制配置（知识库、规则集）
- 每个员工的使用偏好

## 三、飞轮机制（技术实现）

```
第一阶段：基础版上线
  平台预置通用保险知识 → Agent能用

第二阶段：使用积累
  员工日常使用 → 产生大量真实对话和判断
  ↓
  后台分析：哪些回答被采纳？哪些被员工修正？
  ↓
  提取模式：老核保员怎么判断疑难case

第三阶段：知识蒸馏
  优秀实践 → 人工审核 → 封装为新Skill
  ↓
  更新知识库 → 所有Agent升级

第四阶段：飞轮转动
  Agent更强 → 更多员工愿意用 → 更多数据 → 更强Agent
```

**技术实现要点**：
- 对话日志全量记录（结构化存储）
- 自动标记"员工修正过的回答"（说明AI答错了，需要学习）
- 定期跑分析报告：高频问题、错误率、优秀案例
- Skill版本管理：基础版v1 → v2 → v3，可回滚

## 四、部署架构

### 客户侧（保险公司）
```
保险公司提供：
├── 服务器/云实例
├── SSL证书（yixiubx.com 或客户自己的域名）
└── 企微管理员权限（创建bot、配置回调）
```

### 我们部署
```
部署在客户服务器上：
├── nginx（反向代理，接收企微回调）
├── cc-connect（消息桥接）
├── Agent Router（消息路由调度）
├── Agent实例 × N（核保、理赔、客服...）
├── 知识库（RAG向量数据库）
└── 对话日志存储
```

### 我们的云端（可选）
```
中央平台：
├── Agent模板管理（基础版Skill的发布和更新）
├── 飞轮数据分析（跨客户的优秀实践提取）
├── 远程运维监控
└── 客户管理后台
```

## 五、关键技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 大模型 | Claude (Anthropic API) | 能力强、Skill生态好 |
| Agent框架 | Claude Code + Skills | 模块化、可扩展 |
| 消息桥接 | cc-connect | 已验证微信/企微/飞书 |
| 向量数据库 | 待定（Milvus/Qdrant/ChromaDB） | 知识库RAG检索 |
| 消息队列 | Redis / RabbitMQ | 并发控制、排队 |
| 对话存储 | PostgreSQL / MongoDB | 对话历史和日志 |
| 部署 | Docker + Docker Compose | 标准化，方便客户侧部署 |
| 反向代理 | nginx | 企微callback接入 |
